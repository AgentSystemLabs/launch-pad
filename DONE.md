## P0 — config lock relief

After the first successful deploy, the project's **identity/shape** is frozen, but the **operational** fields are now mutable: `cpu`, `memory`, `replicas`, `env`, and `secrets` (key names). Deploy still aborts before build on an identity change (placement, domain, port, build inputs, health check, rollout, add/remove/rename a service) — there is no bypass flag for those.

- [x] **`launch-pad scale replicas <service> <n>`** — edits TOML + runs a single-service deploy; replica count is now a safe post-deploy mutation.
- [x] **`launch-pad scale cpu|memory …`** — wraps the two vertical-scale fields, validated against node capacity (the deploy admission check).
- [x] **Allowlisted post-deploy TOML edits** — `replicas`, `env`, `secrets` keys are mutable; identity fields stay locked (no `--force` escape hatch — re-create the footprint to change identity).
- [x] **`launch-pad config set|unset <service> KEY[=VALUE]`** — ergonomic CLI that edits the `env` table instead of hand-editing a locked file (secrets → `launch-pad secret`).

> Verified by unit tests (`config-lock.test.ts`, `toml-edit.test.ts`, `deploy.test.ts`) and a dedicated real-AWS regression (`pnpm e2e:scale`: provision → deploy worker → scale 1→3 → `config set` env → scale 3→1, with teardown). The full-lifecycle `pnpm e2e` also exercises `scale`/`config` against a web service behind an edge.

## P0 — service & node lifecycle

- [x] **`launch-pad undeploy` / `service remove`** — removes a project footprint (or one `--service`) from each node's `desired.json` (the agent drains the containers), **trims the config baseline** so a follow-up deploy of the edited `launch-pad.toml` passes the lock (whole-footprint undeploy clears the baseline → fresh first deploy), optional `--purge-secrets` (SSM), and waits for drain. Ownership-scoped (never touches another project) with CAS-guarded baseline writes. Unit-tested (`shared/undeploy.test.ts`, `cli/commands/undeploy.test.ts`) + real-AWS regression (`pnpm e2e:undeploy`). ECR images are intentionally kept (immutable, preserve rollback).
- [x] **`launch-pad node evacuate <node>`** — moves the current project's **cluster-placed** services off a node (= `rebalance --drain`), reusing published images, so `node destroy`/`pause`/`resize` (which refuse to orphan scheduled services) accept it. Refuses to evacuate a **pinned** service (config-locked placement) or drain the last app node. Run per project. Real-AWS verified (`pnpm e2e:rebalance`: evacuate node-b → replicas consolidate → `node destroy` succeeds). _Follow-up: cross-project auto-evacuate-all + edge-routing drain for web services._
- [x] **`launch-pad rebalance`** — replans the footprint's cluster-placed services across the current app pool via the same scheduler `deploy` uses, reusing each service's published image (no rebuild), republishing per-node desired.json (gainers before reducers) + cleaning vacated nodes. `--dry-run` / `--drain <node>` / `--env`. Config-lock-safe (toml must match baseline; only placement changes); idempotent ("already balanced" → no write). Pure diff planner (`deploy/rebalance-plan.ts` `diffPlacement`) + extracted `buildCandidateNodes` (shared with deploy) unit-tested; real-AWS regression (`pnpm e2e:rebalance`: deploy even a:2/b:1 → evacuate → spread back). _Follow-up (concurrency review): rebalance is eventually-consistent + not concurrency-fenced against a simultaneous deploy/scale of the same footprint (a re-run reconciles); a footprint version-fence + per-service surge-then-shed ordering + edge-domain shrink are the hardening follow-ups._
- [x] **Destroy should refuse or auto-evacuate** — `node destroy` **refuses** when a node still hosts scheduled services (lists the at-risk `project/service`s), requiring `--force` to orphan them. Unit-tested (`nodesThatWouldOrphan`) + verified in `pnpm e2e:scale`.
- [x] **`node destroy --evacuate` (auto-evacuate inline)** — one command instead of the manual `node evacuate` → wait → `node destroy` two-step: it auto-moves the **current project's cluster-placed** services off the doomed node(s) (= `rebalance --drain`, draining the **whole** target set at once so a replica never lands on a sibling that's also going away), **waits for the surviving pool to converge** (so it never terminates the node before the footprint is confirmed up elsewhere — a stuck convergence aborts with nothing torn down), then tears the node(s) down. Pinned (`node`/`nodes`) services and **other projects'** services can't be auto-moved — those still trip the orphan gate (refuse unless `--force`); a typed `EvacuationBlockedError` (pinned-on-drain / would-leave-no-app-nodes) is caught so the gate decides rather than crashing. `--env` targets a named footprint; `--timeout` bounds the wait (default 300s). Pure planners `assessEvacuation` (movable vs. unmovable per node) + `resolveDrainSet` unit-tested; `runRebalance` gained `drainNodes`/`wait`/`quiet`/`timeout`. Real-AWS regression (`pnpm e2e:destroy-evacuate`: plain destroy refuses → `destroy --evacuate` drains 1 replica onto the other node, waits to 3, destroys it → last-app-node `--evacuate` refuses). _Follow-up: cross-project evacuate-all + edge-routing drain for web services._
- [x] **Non-disruptive vertical scale — `node resize --evacuate`** — plain `node resize` stops the instance under its services; `--evacuate` makes it rolling for the current project's **cluster-placed** services: drain them off the node (= `rebalance --drain`), **wait** for them to be confirmed running elsewhere, resize the emptied instance (stop → retype → start), then **rebalance back** and wait again — a replica is never down with the instance. Refuses up front when non-disruptive is impossible: a **paused** node (nothing running; the rebalance-back would wait on a stopped agent) or a node the project **pins** a service to (`rebalance --drain` hard-blocks on config-locked placement); an `EvacuationBlockedError` from the drain (e.g. last app node) aborts before any downtime. Other projects' services ride the brief stop/start (warned); edge/both ingress still blips while Caddy restarts. `--env` / `--timeout` as in `destroy --evacuate`; the shrink capacity guard still applies (computed pre-drain). Pure planner `planResizeEvacuation` (drain / resize-only / refuse-stopped / refuse-pinned) unit-tested (`resize-evacuate.test.ts`); real-AWS regression (`pnpm e2e:resize-evacuate`: drain observed live mid-command — all 3 replicas running on the survivor **before** the instance stopped — then t3.small→t3.medium retype, even spread restored, paused-node refusal). _Follow-up: `node resize` for pinned services via temporary co-residency; reactive autoscaling policy._

## P0 — capacity & node autoscaling

- [x] **Auto-add app node on capacity pressure** — when cluster-auto-placed services don't fit the current pool, deploy now **adds `app-<n>` node(s)** (sized like the cluster's existing nodes) and re-plans onto the larger pool, instead of erroring "reduce cpu/memory/replicas". Works for both `even` (round-robin overflow detected post-plan) and `capacity` (planner throws a catchable `CapacityPlacementError`) scheduling; bounded by the replica count; spend-gated by the same provision confirmation (`--yes` in CI); `--no-create` restores the hard error. Pure `planClusterPlacementAutoAdd` + `templateCandidateNode` + `nextAppNodeId` + `CapacityPlacementError` in `cli/src/deploy/placement.ts` (reuses the existing planner/provision/sizing — added nodes are provisioned for real, auto-sized to their placement; a non-capacity planner error like "split needs an edge" is rethrown, never triggering an add). Unit-tested (`placement.test.ts` +10) + real-AWS regression (`pnpm e2e:auto-add`: bootstrap 1 node at replicas=1 → `scale replicas 3` auto-adds `app-2` → 3 replicas spread 2/1).
- [x] **Reactive autoscaling policy — `launch-pad autoscale`** — declarative min/max app nodes + live CPU/memory utilization thresholds, stored per cluster (`autoscale` in `cluster.json`), applied by a one-shot **reconcile pass** (`autoscale run`, cron-able) instead of a daemon — no control plane. Each agent now embeds its latest **host utilization sample** in `status.json` (`host`: cpu %, memory used/total, `sampledAt`; excluded from the write-on-change fingerprint so no PUT storm), so the CLI reads live utilization straight from S3 — no CloudWatch, no new IAM. One action per pass: **scale-out** (minNodes floor — bypasses cooldown — or avg utilization ≥ threshold; provisions the next `app-<n>` sized like the largest pool node, role `app` behind a dedicated edge else `both`, then rebalances onto the new pool, capped at maxNodes) or **scale-in** (every node below the threshold AND the survivors can absorb the victim's **reserved** footprint — live utilization never overrides reservation admission, a gap the e2e caught live when the drain publish hit `assertCapacity`; drains the least-utilized feasible non-protected node via `rebalance --drain`, **waits for convergence**, then gives the victim's agent a drain-grace window (graceful stop + upstream-shard retraction) before terminating — refuses the cluster's default edge, any node still hosting another project's / a pinned service after the drain, and any pass where a pool node lacks fresh metrics). Thrash/race guards: `scale-in % < scale-out %` (schema-enforced), cooldown between utilization actions, the pass **CAS-claims `lastScaleAt`** (conditional PUT) *before* acting so overlapping cron fires abort cleanly and a crashed pass stays cooled down, a still-provisioning node holds its pool slot (no double scale-out), `--yes` required for billable/destructive actions in `--json`/cron mode. Pure planner `planAutoscale` + `scaleOutNodeSpec` + policy schema in `shared/src/autoscale.ts` (unit-tested, 25 cases); real-AWS regression (`pnpm e2e:autoscale`: BURN-toggled memory-hog worker — memory, not CPU, because t3 standard-mode credit throttling caps a busy loop at the ~20%/vCPU baseline → idle no-op → hot pool provisions `app-2` → holds at max → replicas spread onto the new node → cold pool drains `app-2` with the replica observed moved to `app-1` **before** teardown → holds at min). Host samples are schema-bounded (cpu 0–100, finite, non-negative memory) so a single compromised node can't skew the pool average into a spend amplifier; min/max nodes are capped at 100 (spend-typo guard).
- [x] **Empty-cluster bootstrap** — the first deploy of cluster-auto-placed services to a cluster with **no nodes** now auto-provisions a single co-located node (`app-1`) and places onto it (parity with the default cluster's single-node auto-provision), instead of erroring "no app nodes". Spend-gated (confirm / `--yes` / `--no-create`); the instance is auto-sized to fit and any edge a `split` service needs is provisioned too. Implemented as a synthetic candidate node (`bootstrapCandidateNode` in `cli/src/deploy/placement.ts`) injected into the planner pool when empty, so all topology/edge/sizing logic reuses the existing planner + provision flow. Unit-tested (`placement.test.ts` — both-role pool eligibility, large-demand placement, co-located/split routing, split-without-edge still rejected) + real-AWS regression (`pnpm e2e:empty-cluster`: empty cluster → `--no-create` refused → deploy bootstraps `app-1` → worker runs → idempotent re-deploy).

## P0 — workers

- [x] **Worker scheduling (cron)** — `cron = "*/5 * * * *"` on a worker `[[service]]` makes it a **scheduled job**: the agent runs **one container per fire** (UTC, standard 5-field vixie semantics incl. lists/ranges/steps and the dom-OR-dow rule) with `--restart no` and judges it by **exit code** — no long-running process, no healthCheck, `replicas` fixed at 1. No overlap (an in-progress run suppresses the next fire), missed fires collapse to the **latest** one (no catch-up storm), and a new schedule is **anchored at first sight** (never replays history); the last fire is recorded crash-safe twice (run-container `launchpad.cronFire` label + agent `state.json`), written *before* the container starts so a crash skips, never duplicates. Status grows an additive `cron` rollup (`lastRunAt`/`lastExitCode`/`nextRunAt`); an armed schedule reports state `running` and deploy/rebalance watch it with `expectedReplicas = 0`, so a cron service can never hang a deploy — and a *failed run* surfaces via exit code/message, never state `error`. The expression is **config-locked identity**; `scale replicas` refuses cron services. Zero-dep evaluator in `shared/src/cron.ts` (16 unit tests) + planner (`planCronService`, 9) + status (6) + schema (10) tests; new fixture `examples/cron-task`; real-AWS regression (`pnpm e2e:cron`: deploy converges at 0 replicas → first fire exit 0 → second fire advances `lastRunAt` → lock refuses a cadence change → undeploy cleans up). Security + concurrency audits run; all 4 actionable findings fixed: never-firing expressions (Feb 30) rejected at parse (would have burned agent CPU every tick), a fire elapsing during a long run is **recorded as skipped** (`cronSkip` — never a back-to-back catch-up execution on exit), a failed state persist now refuses to start the run (record-before-run stays a hard guarantee, logged), and anchor seeding only trusts a **running** container's fire label (a stale exited leftover can't trigger a catch-up run after undeploy→redeploy). _Follow-ups: `cron.lastExitCode` in `alerts check`; timezone support; jitter._

## P0 — developer experience

- [x] **Preview environments (`deploy --env <env> --ttl <duration>` + `launch-pad preview`)** — the full PR lifecycle on top of the existing `--env` flag. An `--env` deploy now **auto-creates a DNS-only Route53 A record** for each env-projected domain at its fronting node's EIP (written BEFORE the convergence wait so propagation/cert issuance overlap it) and writes a **preview marker** (`projects/<project>-<env>/preview.json`, `PREVIEW_MARKER_VERSION` — CLI-only state, the agent never reads it) recording the env's domains, the exact DNS records its deploys created, `createdAt`, and the `--ttl` expiry (`30m`/`72h`/`7d`, bounds 1m–90d; a re-deploy with `--ttl` re-arms the deadline, without keeps it; marker DNS records merge by name so a partial deploy can't orphan one at teardown). Auto-DNS **never clobbers a foreign record**: it writes only when the record is absent, already correct, or named in the env's own prior marker (so a node replacement re-points it) — anything else is skipped with a pointer at the explicitly-confirmed `dns setup`. `launch-pad preview list` enumerates the cluster's previews (expiry + domains); `preview destroy <env>` undeploys the whole env footprint (reusing the ownership-scoped `planUndeploy`/`applyUndeploy`, so co-located siblings of OTHER footprints are untouched), waits for the drain, deletes **exactly** the marker-recorded Route53 records (`--keep-dns` skips), and sweeps the footprint's `projects/` state; `preview prune` is one cron-able pass (no daemon, same model as `autoscale run`) that destroys every expired preview — `--dry-run` previews, a failed teardown keeps its marker so the next pass retries, and `--json` mode **requires `--yes`**. Only marker-backed envs are eligible — the base project's footprint can never be destroyed by `preview`. **Security-hardened against a tampered marker** (the marker lives in S3 and drives destructive actions, so it is never trusted alone): the schema shape-pins every field that reaches a destructive path (env/project as DNS labels, domains/record names as hostnames, IPv4 `ip`, Route53 `zoneId` form, owner charset — also guarding the `projects/<owner>/` prefix sweep against path escape); destroy cross-checks the marker's base project against the deploy-written **config baseline** and fails closed on mismatch or an unreadable baseline (a forged `project "my" + env "app"` marker can't destroy a real `my-app` production footprint); and every DNS delete is **verified live first** — the record must still point at exactly the marker's IP or it's reported and left alone (no S3-write → Route53-delete escalation, no false "deleted" for a drifted record). `preview destroy`/`prune` in `--json` mode both require `--yes`. Pure planners in shared (`preview.ts`: marker schema with owner=project+env cross-check, `parsePreviewTtlMs`, `buildPreviewMarker`, `isPreviewExpired` — fail-closed on an unparsable deadline, `planPreviewPrune`, `selectPreviewMarkers`) + `cli/src/dns/auto.ts` (`ensurePreviewDnsRecords` ownership rules) unit-tested (+23); **real-AWS regression** (`pnpm e2e:preview`: `deploy --env pr-a --ttl 1m` converges + the projected A record exists in Route53 with NO `dns setup` run → second env without TTL → `preview list` shows both → unconfirmed JSON prune refused → after the TTL lapses `prune --yes` destroys ONLY pr-a — containers stopped, A record gone, marker swept, pr-b still running with its record intact → `preview destroy pr-b` → final prune is a clean no-op → cluster destroy with zero warnings).
- [x] **Unified app teardown — `launchpad destroy` (replaces `undeploy` + `preview`)** — one symmetric mental model: `deploy` puts an app up, `destroy` takes it down; named environments use the same `--env` flag on both. `destroy` (no flags) removes the base footprint (former `undeploy`); `destroy --env <name>` is the full marker-driven env teardown (undeploy + auto-created Route53 records unless `--keep-dns` + `projects/` state sweep — works without a launch-pad.toml in cwd); `destroy --list-envs` and `destroy --prune-expired [--yes]` replace `preview list`/`prune` (no `--yes` = dry run; `--json` still requires `--yes`). Clean break, no deprecated aliases; infra teardown stays `node destroy`/`cluster destroy`. Internal names kept (shared `undeploy.ts`/`preview.ts` planners, `preview.json` marker key — internal S3 contract, never user-facing). e2e renamed: `pnpm e2e:destroy` / `pnpm e2e:destroy-env`.

## P1 — CI/CD

- [x] **Remote build (`deploy --remote-build`)** — deploy without local Docker, for slim CI runners with no docker daemon. Per service, deploy packs the build context into a tarball, uploads it under the footprint's `builds/` S3 prefix (deleted after the build), and runs one build in a **per-cluster CodeBuild project** (`launch-pad-build-<cluster>`, privileged Linux small, `NO_SOURCE` — the buildspec pulls the tarball itself) producing the **same immutable content-addressed linux/amd64 tag** the local buildx path would; everything after the build is identical, and an image already in ECR skips its build. `.dockerignore` is honored for the **upload** (literals + root globs like `*.pem`/`.env*` + `**/` any-depth forms — so secret-guarding globs keep files out of S3; negations conservatively disable the exclusions they re-include under, and the dockerfile/`.dockerignore` always ship since docker reads both out-of-band; the full file ships so docker re-applies every pattern remotely). First use creates the project + a least-privilege service role (`launch-pad-codebuild-<cluster>`: ECR push, **own-cluster** `builds/*` read only — never desired/status state or another cluster's contexts — own log group); `cluster destroy` removes project + role + log group. Push runs in the **same buildspec phase** as the build so a failed build never cascades a doomed push; the build **retries 3× with backoff** (CodeBuild NAT IPs get 429'd by Docker Hub for anonymous pulls — docs recommend `public.ecr.aws/docker/library/*` base images); failures print the failing command's log context (not the useless literal tail). Dockerfile must live inside its `context` (validated before any AWS call); mutually exclusive with `--restart`/`--image`. Operator IAM policy extended (CodeBuild scoped to `launch-pad-build-*`, PassRole pinned to codebuild.amazonaws.com, log read/delete on the build groups only). Pure planning (`cli/src/deploy/remote-build.ts`: names, trust/service policies, buildspec, dockerfile-in-context, dockerignore matcher) + packer (`context-pack.ts`) unit-tested; **real-AWS regression** (`pnpm e2e:remote-build`: local docker **shimmed to a failing stub on PATH** for every CLI call → v1 builds on CodeBuild + converges → project exists (privileged, NO_SOURCE) → unchanged re-deploy skips the build with no container churn → content change builds v2 remotely and rolls → destroy removes project + role + log group). The live run caught two real bugs unit tests couldn't: a `.dockerignore` that lists `Dockerfile` (legal — docker reads it out-of-band) broke the remote build until the packer force-included it, and Docker Hub 429s motivated the retry + base-image guidance.

- [x] **Official GitHub Action** — `launch-pad setup github-oidc --repo <owner/name>` emits a keyless (OIDC) `.github/workflows/deploy.yml`: checkout → assume role via OIDC → Docker Buildx → `npx @agentsystemlabs/launch-pad deploy --yes`. Now **concurrency-guarded** (one deploy per ref, cancel superseded — matches deploy's CAS write protection) with a manual `workflow_dispatch` trigger, and documented in [cli.md](docs/cli.md#setup-github-oidc) with **caching guidance** (pin the CLI version, optional `cache: npm` when a lockfile exists, Dockerfile layer ordering, self-hosted runner for heavy builds). Pure `buildDeployWorkflow` unit-tested (+2); the deploy-under-OIDC runtime is already real-AWS-verified by `pnpm e2e:operator-iam` (full provision→deploy→destroy under a scoped role).
- [x] **`deploy --image <uri>`** — `deploy --service <name> --image <uri>` skips the build and redeploys an existing immutable ECR tag (rollback / promote). Validated to the service's own repo + an existing tag; re-rolls in place health-gated; idempotent on a repeat. Pure parser (`shared/src/ecr.ts` `parseEcrImageUri`) + `loadOverrideImage` unit-tested; real-AWS regression (`pnpm e2e:deploy-image`: v1 → v2 → roll back to v1 without building). Next: a `launch-pad rollback` wrapper that auto-picks the previous tag (ECR push order / deploy history).
- [x] **Monorepo `deploy --changed <ref>`** — first-class "deploy changed services only": a `git diff` against `<ref>` (committed **and** uncommitted **and** untracked changes, since all three land in the image a rebuild would push) maps to the services whose docker **build inputs** — their `context` directory or `dockerfile` — changed, and deploys only those. With nothing changed it's a clean exit-0 **no-op** so a docs-only CI commit doesn't fail the deploy job. Wire it as `launch-pad deploy --changed origin/main --yes` (or `${{ github.event.before }}`). Mutually exclusive with `--service`/`--image`/`--restart`. **Depends on a partial-deploy correctness fix:** a subset deploy (`--changed`, `--service`, and the `scale`/`config set` edits that wrap a single-service deploy) now **upserts** each node's `desired.json` (`mergeProjectServicesPartial`) instead of replacing the project's whole footprint — so (re)deploying one service no longer drops its **co-located siblings** (a latent, data-loss-class bug the old full-replace merge had: `scale web` on a node also running `worker` removed `worker`). **Hardened:** `<ref>` is resolved to a commit SHA before diffing, so it can never be a git flag (`--output=…`) or a gitrevision expression (`HEAD:path`, `@{upstream}`) that shifts the diff base — safe when CI interpolates an untrusted ref. Pure mapping (`cli/src/deploy/changed-services.ts` `selectChangedServices`/`collectChangedPaths`) + `mergeProjectServicesPartial` (shared) + `publishDesired(partial)` / `capacityDemands(partial)` unit-tested; **real-AWS regression** (`pnpm e2e:deploy-changed`: co-located api+worker on one node → a partial `scale worker` deploy **preserves** `api` byte-for-byte → editing only `apps/worker` + `deploy --changed <v1>` rolls **only** `worker` and leaves `api`'s container + published image untouched → `deploy --changed HEAD` is a clean no-op).

## P1 — rollback & release safety

- [x] **`launch-pad rollback`** — `rollback --service <name> [--to <tag>]` redeploys a service's previous immutable ECR build (or a specific tag) without rebuilding, by delegating to `deploy --image`. Auto-pick is "most-recent build pushed before the current one" (`findPreviousImageTag` over `listRepoImageTags`); refuses to silently roll forward (asks for `--to` when nothing older). Pure picker + `resolveRollback` unit-tested; real-AWS regression (`pnpm e2e:rollback`). _Follow-up: whole-footprint rollback once deploy history (`events/`) lands — today rollback is per-`--service`._
- [x] **Deploy history** — every `deploy` appends an append-only event to `…/projects/<footprint>/events/` in S3 (who=caller ARN / when / image per service / kind / converged); `launch-pad history [--service] [--env] [--limit]` reads them newest-first. Advisory (never read by the agent), best-effort (a failed write can't fail a deploy), no secret values. Schema in `shared/src/events.ts`; recorded by `deploy`'s `recordDeployEvent`. Unit-tested (`events.test.ts`, `history.test.ts`, `s3-keys.test.ts`) + real-AWS regression (`pnpm e2e:history`).

## P1 — observability & alerting

- [x] **Alerting** — `launch-pad alerts check` is a scheduled-probe health check (cron / GitHub Action) that scans the cluster and flags **real faults on nodes that should be running**: a live node whose agent stopped heartbeating (or, past a ~10-min boot grace, never reported) and a service in `error` / fully down (0 running replicas). Deliberately quiet on non-faults (a paused node's agent is off on purpose; a still-booting node gets its grace; a partial mid-rollout replica dip doesn't alert). Exits **non-zero** on any alert (CI-gateable) and POSTs a Slack/Discord-compatible payload to `--webhook` (or `LAUNCHPAD_ALERT_WEBHOOK`). Pure `evaluateAlerts` + `buildAlertPayload` (`cli/src/alerts/`) unit-tested (15) + real-AWS regression (`pnpm e2e:alerts`: deploy a worker → clean check → **terminate the instance out-of-band** so the registry still reads "up" while the agent goes silent → alert fires + webhook delivered + non-zero exit). _Follow-up: continuous (not on-demand) monitoring, cert-renewal-fail + node-drift signals, and a deploy-convergence-fail hook (deploy already exits non-zero on non-convergence, which is CI-gateable today)._
- [x] **Dashboard production story** — documented the gap in [`docs/dashboard.md`](docs/dashboard.md#production-story-the-gap): the dashboard is a local operator convenience (localhost-only, no auth, drives the CLI with your local AWS creds), **not** a hosted control plane — and a hosted one is deliberately out of scope (Launch Pad is declarative with no control-plane server). Documents the supported shared-operation paths instead: CI/CD (OIDC) as the shared control surface, scheduled `alerts check`/`cost --budget` gates with webhooks, or running the dashboard behind your own auth/SSH tunnel.

## P1 — data & stateful apps

- [x] **Persistent volumes** — `[[service.volumes]]` (`name` + container `path`) declares a node-local docker named volume the agent mounts into the service's container(s); its data **survives a container replace** (rolling deploy / `deploy --restart` / reboot) so SQLite, uploads, and local caches don't reset on every deploy. A volume-bearing service must be **pinned to a single `node`** (cluster auto-placement / `nodes` would move or split the data) and volumes are **config-locked identity** (no add/remove/re-path after the first deploy). The volume name is per-service + index-independent (`launchpadvol_<project>_<service>_<name>`) so it's re-mounted across the index-renumbering rollout. Deploy **refuses** to publish a volume-bearing service onto a legacy rust-agent node (`assertVolumesSupported`) rather than silently drop the mount. Schema in `shared/src/config.ts` (`VolumeDeclSchema` + validation) + `desired.ts` (wire) + `config-lock.ts` (locked); mounts in `agent/src/docker.ts` (`buildRunArgs`/`volumeName`). Unit-tested (config +8, config-lock +3, agent docker +4, deploy +7) + real-AWS regression (`pnpm e2e:volumes`: deploy a `/data` worker → boot count 1 → `deploy --restart` replaces the container → boot count 2 (data persisted) → config lock refuses a volume-path change). EBS-volume attach (cross-node-failure durability) is a follow-up; named volumes give container-replace durability on the node's root EBS.
- [x] **Backup/restore** — `launch-pad backup` exports a cluster's authoritative S3 state (registry `cluster.json`/`node.json`, `desired.json`/`status.json`, config baselines, deploy events) to a local directory keyed by S3 key + a `manifest.json`; `launch-pad restore <dir>` re-uploads it (gated by a confirm / `--yes`). Read-only backup; state-only (no plaintext secrets — `desired.json` carries SSM refs, not values). Restore fails closed against a tampered backup dir: three pre-AWS guards (clean relative key / within the target cluster's keyspace / present in the manifest) + a per-file size cap, and symlinks are skipped. Default cluster sweeps the legacy `nodes/`+`projects/` roots; a named cluster sweeps `clusters/<id>/`. Pure planner (`cli/src/backup/plan.ts` — `backupPrefixesForCluster`/`isSafeBackupKey`/`keyUnderPrefixes`) unit-tested; security-reviewed (traversal guards confirmed sound; manifest-enforcement + size-cap hardening added). Real-AWS regression (`pnpm e2e:backup`: set up cluster + synthetic object → backup → delete from S3 → restore → assert byte-for-byte → cluster resolves again). ECR images are out of scope (immutable, rebuildable).
